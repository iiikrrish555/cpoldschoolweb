---
name: Porting legacy Python2 + MySQL Flask apps to Replit
description: Patterns for bringing old Flask/SQLAlchemy/MySQL codebases (originally Py2.7) onto Python 3 + Replit Postgres.
---

When importing an old Flask app that targets MySQL and Python 2.7:

- `flask_sqlalchemy_session` is abandoned and imports `_app_ctx_stack`, which no longer exists in modern Flask (2.2+). It cannot be installed as-is. Replace it with a plain SQLAlchemy `scoped_session(sessionFactory)` plus a `@app.teardown_appcontext` handler that calls `session.remove()`.
  **Why:** the package import fails outright on any current Flask version, not just a deprecation warning.
  **How to apply:** anywhere the app wires up `flask_scoped_session(sessionFactory, app)`.

- If each SQLAlchemy model file calls its own `declarative_base()`, foreign keys referencing tables defined in *other* model files fail with `NoReferencedTableError` — declarative_base() metadata is per-call, and FK resolution needs all referenced tables in the same `MetaData`.
  **Why:** SQLAlchemy can't resolve `ForeignKey('othertable.id')` across separate metadata registries.
  **How to apply:** consolidate all models onto one shared `Base` (single `declarative_base()` in a common module, imported everywhere) before calling `Base.metadata.create_all(engine)`.

- MySQL DDL like `server_default=text("current_timestamp()")` (with parens) is not valid Postgres DDL syntax for a bare current-timestamp default; use `text("current_timestamp")` (no parens) when porting MySQL server_default expressions to Postgres.

- `Column(nullable=..., index=...)` with no type argument is a common latent bug in old MySQL-autogenerated models (works under sqlacodegen but errors when SQLAlchemy tries to build DDL). Add an explicit `Integer`/`ForeignKey` type when porting.
